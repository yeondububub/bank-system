# Bank-System Backend - Step 5: Redisson을 활용한 Redis 분산 락(Distributed Lock) 도입

## 1. 개요 및 배경 (Context)
이전까지 도입했던 데이터베이스 비관적 락(Pessimistic Lock)은 데이터베이스 트랜잭션이 시작된 후 락을 획득하고, 트랜잭션이 커밋될 때까지 계속해서 락과 데이터베이스 커넥션을 점유합니다.
결제 승인과 같이 처리 시간이 비교적 긴 비즈니스 로직이나 고트래픽 상황이 발생하면, 많은 스레드가 DB 커넥션을 붙잡은 채 대기하는 **Connection Starvation(커넥션 풀 고갈)** 현상이 발생할 위험이 있습니다.

또한 여러 대의 애플리케이션 서버로 스케일아웃(Scale-out)된 환경에서 데이터베이스에 가해지는 동시성 제어 부하를 줄이고, 비즈니스 진입 전에 임계 영역(Critical Section)을 안전하게 잠글 분산형 락 메커니즘이 요구됩니다.

## 2. 결정 사항 (Decision)
인메모리 데이터 구조 저장소인 Redis와 **Redisson** 라이브러리를 활용하여 **분산 락(Distributed Lock)** 시스템을 도입하고, 기존 DB 비관적 락을 점진적으로 대체하도록 설계합니다.

1. **Redisson 펍-섭(Pub-Sub) 기반 락 획득**:
   Redisson은 스핀 락(Spin Lock) 방식이 아닌 펍-섭 메커니즘을 사용하므로, 락이 해제되었을 때 대기 중인 스레드에 알림을 주어 Redis 서버에 과도한 부하(CPU 점유율 상승)를 가하지 않고 효율적으로 대기할 수 있습니다.
2. **트랜잭션 바운더리보다 넓은 락 라이프사이클 (AOP 도입)**:
   트랜잭션 커밋(DB 반영)이 완전히 끝나기 전에 분산 락이 해제되면, 다른 대기 중인 스레드가 락을 획득하여 아직 DB에 반영되지 않은 옛날 데이터를 읽어가서 정합성이 깨질 수 있습니다. 
   이를 해결하기 위해, 트랜잭션이 시작하기 전에 락을 획득하고 트랜잭션이 완벽히 커밋/롤백된 이후에 락을 반납하도록 설계해야 합니다.
   이를 선언적이고 간결하게 구현하기 위해 `@DistributedLock` 커스텀 어노테이션과 `DistributedLockAop` (Aspect)를 구현하여 트랜잭션(`@Transactional`) 바깥쪽에서 락 라이프사이클을 통제합니다.
3. **로컬 개발 환경 설정**:
   Redis 연동을 위해 `docker-compose.yml`을 작성하여 로컬 MySQL과 Redis 컨테이너를 한 번에 띄울 수 있도록 인프라 구성을 추가합니다.

## 3. 결과 및 영향 (Consequences)
- **장점**:
  - 데이터베이스의 로우 락 및 커넥션 점유 대기 시간을 크게 줄여주므로 애플리케이션의 고가용성 및 처리량이 크게 개선됩니다.
  - 공통 모듈에 AOP 방식으로 작성하여, 결제 외에도 동시성 제어가 필요한 다른 도메인 메서드에 어노테이션 선언만으로 손쉽게 적용할 수 있습니다.
- **단점/고려사항**:
  - Redis가 싱글 포인트 오브 페일러(SPOF)가 될 수 있으므로, 실 서비스 배포 시 Redis Sentinel이나 Redis Cluster와 같은 고가용성(HA) 구성이 필수적입니다.
  - 락 점유 타임아웃(leaseTime) 내에 처리가 안 끝날 경우 락이 자동 해제되어 동시성 제어가 무력화될 수 있으므로, 비즈니스 시간의 임계값을 신중히 튜닝해야 합니다.
