# Bank-System Backend - Step 2: 비관적 락(Pessimistic Lock)을 이용한 결제 승인 동시성 제어

## 1. 개요 및 배경 (Context)
한 개의 주문번호(`orderId`)에 대해 네트워크 지연이나 클라이언트의 다중 클릭 등으로 인해 여러 건의 결제 승인 요청이 거의 동시에 발생할 수 있습니다. 
만약 동시성 제어를 하지 않는다면, 여러 요청이 한꺼번에 조건 검증(예: 결제 완료 상태가 아닌 것 확인)을 통과하여 동일 결제 건에 대해 이중 승인이 일어나거나 데이터 정합성이 깨질 수 있습니다.

## 2. 결정 사항 (Decision)
가장 보수적이고 확실하게 데이터의 정합성을 지키기 위해 데이터베이스 수준의 **비관적 쓰기 잠금(Pessimistic Write Lock)** 을 도입합니다.

1. **DB 잠금 적용**:
   `PaymentJpaRepository` 인터페이스에 `@Lock(LockModeType.PESSIMISTIC_WRITE)` 어노테이션이 선언된 `findByOrderIdWithLock` 메서드를 추가합니다. 이를 통해 결제 승인 비즈니스 로직 진입 전에 해당 결제 로우(Row)에 락을 획득합니다.
2. **트랜잭션/락 라이프사이클 분리 (`PaymentFacade` 도입)**:
   도메인 서비스(`PaymentService`)는 순수한 도메인 로직만 담당하게 하고, 기술적 트랜잭션 바운더리 및 락의 수명을 오케스트레이션하기 위해 `module-api`에 `PaymentFacade` 파사드 계층을 도입합니다.
   `PaymentFacade.approvePayment`에 `@Transactional`을 선언하여 **락 획득 ➔ 도메인 비즈니스 로직(승인 상태 변경) ➔ 데이터베이스 저장 및 커밋**이 하나의 트랜잭션 안에서 수행되도록 보장합니다.

## 3. 결과 및 영향 (Consequences)
- **장점**:
  - 데이터베이스의 강력한 쓰기 락을 사용하므로, 분산 서버 구조에서도 정합성을 확실하게 보장하며 데이터 수정 유실을 방지합니다.
- **단점/고려사항**:
  - 락을 보유한 채로 오랜 비즈니스 로직을 타게 되면 데이터베이스 커넥션이 고갈되거나 처리량(Throughput)이 하락할 위험이 있습니다.
  - 추후 외부 API 연동 등 지연이 긴 작업이 발생할 경우 락의 유지 범위를 리팩터링해야 합니다.
