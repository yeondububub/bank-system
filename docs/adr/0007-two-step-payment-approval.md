# 0007. 2단계 결제 승인 프로세스 및 대사(Reconciliation) 시스템 도입

## 1. 배경 (Context)
기존의 결제 승인(`PaymentFacade.approvePayment`) 로직은 계좌 잔액 차감과 외부 PG사 승인 API 호출(`pgPort.pay`)을 하나의 `@Transactional` 어노테이션이 지정된 단일 트랜잭션 내에서 동기식으로 실행했습니다. 이 구조는 다음과 같은 한계와 취약점이 존재합니다:

1. **데이터베이스 커넥션 점유 (Connection Starvation)**
   - 외부 PG사의 API 호출은 네트워크 통신을 포함하므로 응답 지연(수백 ms ~ 수 초)이 발생할 수 있습니다. 
   - 이 기간 동안 DB 커넥션 및 행 락(Row Lock)을 계속 유지하고 있게 되어, 순간적으로 결제 트랜잭션이 몰리면 커넥션 풀이 고갈되어 전체 서비스가 마비될 위험이 큽니다.
2. **분산 데이터 상태 불일치 (Distributed Data Inconsistency)**
   - 외부 PG사 승인은 성공했으나, 로컬 DB 커밋 시점에 일시적인 네트워크 장애나 서버 OOM, 데드락 등으로 트랜잭션이 롤백되는 경우, **고객은 외부에서 실결제 되었으나 로컬 DB에는 결제 실패/대기 상태로 남아 상품 구매가 누락되는 심각한 오류**가 발생할 수 있습니다.

## 2. 결정 (Decision)
이 문제를 해결하기 위해 외부 네트워크 호출과 내부 DB 트랜잭션을 철저히 분리하는 **2단계 승인 프로세스(2-Step Approval)**를 도입하고, 장애 시 최종 정합성을 보장하는 **대사(Reconciliation) 시스템**을 구축하기로 결정했습니다.

### 2.1 2단계 승인 프로세스 설계 (PaymentFacade)
결제 승인 로직을 트랜잭션 단위로 쪼개어 동기식 외부 호출이 로컬 DB 트랜잭션 외부에서 실행되도록 오케스트레이션합니다.

1. **상태 확장**: 결제 상태에 `APPROVING` (승인 진행 중) 상태를 추가합니다.
2. **1단계: 준비 단계 (`prepareApproval`)**
   - `@Transactional`을 적용하여 원자적으로 실행.
   - 결제 상태를 `PENDING`에서 `APPROVING`으로 전환.
   - 계좌 잔액에서 미리 결제 금액만큼 **선출금(잔액 차감)** 처리하여 타 결제 요청에 의한 중복 출금(Double Spending)을 사전에 차단합니다.
3. **외부 PG 호출**
   - DB 트랜잭션 바운더리 **외부**에서 `pgPort.pay`를 호출합니다. 이 동안 DB 커넥션은 점유되지 않습니다.
4. **2단계: 최종 처리 단계**
   - **성공 시 (`completeApproval`)**: 결제 상태를 `APPROVING` ➔ `SUCCESS`로 변경하고 완료 이벤트를 발행합니다.
   - **실패 시 (`failApproval`)**: 결제 상태를 `APPROVING` ➔ `FAILED`로 변경하고, 선출금했던 금액을 다시 고객 계좌로 **재입금(보상 트랜잭션)** 처리합니다.

*참고: 전체 흐름을 감싸고 있는 `approvePayment` 메소드 상위에는 `@DistributedLock`을 유지하여, 동일 주문에 대한 동시 승인 요청을 차단합니다.*

### 2.2 대사(Reconciliation) 시스템 도입
외부 PG사 승인 API 호출 직전 또는 호출 중 애플리케이션 서버가 비정상적으로 종료되는 경우, 데이터는 `APPROVING` 상태에 멈춰 있고 고객의 돈은 선출금된 상태로 남아있게 됩니다.

1. **대사 스케줄러 (`ReconciliationScheduler`)**
   - 백그라운드에서 주기적으로 `APPROVING` 상태의 결제 건 중 최종 수정 시간(`updatedAt`)이 일정 시간(예: 10초) 경과한 대상을 수집합니다.
2. **외부 상태 조회 API 연동**
   - 외부 PG사에 결제 상태 조회 API(`GET /v1/payments/{orderId}`)를 호출하여 승인이 실제로 성공했는지 조회합니다.
   - **성공 확인 시**: `completeApproval`을 실행하여 결제를 `SUCCESS`로 확정합니다.
   - **실패/기록 없음 확인 시**: `failApproval`을 실행하여 결제를 `FAILED`로 변경하고 계좌에 금액을 환불(보상 트랜잭션)합니다.

---

## 3. 결과 (Consequences)
* **장점**:
  - 외부 통신 시간이 로컬 DB 트랜잭션 범위 밖으로 분리되어, 느린 외부 PG 호출 상태에서도 DB 커넥션을 점유하지 않아 고가용성이 보장됩니다.
  - 네트워크 단절이나 갑작스러운 장애 발생 시에도, 백그라운드 대사 작업이 동작하므로 결제 데이터 정합성이 결국 일치(Eventual Consistency)하게 됩니다.
* **단점 및 단점 극복**:
  - 기존 1회 DB 트랜잭션에서 2회(준비, 확정)로 분할되어 DB IO 횟수가 증가합니다. 하지만 외부 네트워크 호출 지연 동안 커넥션을 점유하는 비용보다 훨씬 저렴하고 안정적입니다.
  - PG사에 상태 조회 API 연동이 필요하므로 PG 연동 기능 요건이 증가합니다. Mock PG 서버에 조회 기능을 추가하여 테스트 환경을 보강합니다.
